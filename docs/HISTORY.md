# 開発の経緯

claude-signal-tray は、複数のアプローチを検証・試行錯誤した末に現在の方式（プロセスI/O量ベースの状態推定）にたどり着いている。今後の変更や他環境への移植を検討する際の参考になるよう、検討した各アプローチと結果を記録する。

## 課題

Claude Cowork はバックグラウンドでタスクを実行するが、以下のような状況になったことに気づきにくいという課題があった。

- 処理が完了しているのに気づかない
- AskUserQuestion 等でユーザーの確認・入力待ちになっているのに気づかない（ウィンドウが裏に隠れている場合など）

Windows標準の通知機能で解決できないか検討したが、利用環境によっては通知機能自体を無効化しているケースがあり、この前提には頼れないことが分かった。

## 検討したアプローチと結果

### 1. status.json ファイル + Cowork「グローバル指示」方式（不採用）

Claude Cowork の「グローバル指示」（Settings > Cowork）に、状態が変わるタイミングで `status.json` というローカルファイルに書き込むよう指示文を設定し、タスクトレイアプリがそれをポーリングする方式。トレイアプリ自体は正常に動作したが、Claudeに指示を守らせてファイルを自動更新させる部分が機能しなかった。

- 単純な質問応答タスクで検証 → 書き込みは一度も行われなかった
- 指示文を「AskUserQuestionの直前は必ず書く」という、より明確で限定的なトリガーに書き換えて再検証 → それでも書き込みは一度も行われなかった

グローバル指示が保存時点で正しい内容になっていることも確認済みで、指示文の書き方の問題ではないと判断した。指示文をどれだけ強く・明確に書いても、Claudeが確実にそれに従ってファイルを更新するとは保証できないという結論に至った。詳細は [docs/global-instructions.md](global-instructions.md) を参照。

### 2. Windows UI Automation による画面監視方式（不採用）

外部から能動的にClaude Desktopの画面内容を監視し、AskUserQuestion等の表示を検知できないか検討した。Windows UI Automation (UIA) はフォーカスを奪わずに他アプリのUI要素を読み取れる仕組みであり、Accessibility Insights for Windows での調査では内容が読み取れることを確認できた。

そこで診断用コンソールアプリ（[experiments/uia-probe](../experiments/uia-probe)）を自作し、`AutomationElement.FromHandle` とツリー走査でClaude Desktopのウィンドウ内容を取得できるか検証したところ、取得できるのはOSのウィンドウ枠に相当する12個程度の汎用要素のみで、実際のチャット内容は一切取得できなかった。これはウィンドウの最小化状態が原因ではなく、Claude DesktopがElectron/Chromiumベースであり、既定では外部のUIAクライアントに対してアクセシビリティツリーを公開していないことが原因と判明した。この制約を回避する現実的な方法がなかったため不採用とした。

### 3. ネットワークパケットキャプチャ方式（検討の末、簡易プローブへ方向転換）

通信の中身はTLSで暗号化されているため、内容を直接解析するには証明書を独自発行してMITM方式で復号する必要があり、侵襲的でリスクが高い（証明書ピンニングがあれば通信自体が壊れる可能性もある）ため採用を見送った。代替として、通信の中身は見ずに、Claudeプロセスの入出力（I/O）量の変化パターンだけから状態を推定できないか、という仮説を立てて簡易な検証（プローブ）に進んだ。

### 4. プロセスI/O量ベースの状態推定方式（採用・現行方式）

Windowsの `PerformanceCounter`（`Process` カテゴリの `IO Data Bytes/sec`）を用いて、`claude` という名前のプロセス全体のI/O量を継続的にサンプリングし、値の水準からアイドル／確認待ち／作業中を推定する方式。診断用コンソールアプリ（[experiments/network-probe](../experiments/network-probe)）による3回の手動検証で、以下の傾向が一貫して確認された。

| 状態 | I/O量の傾向 |
| --- | --- |
| 完全アイドル | ほぼ0（単発の突発スパイクを除く） |
| AskUserQuestion表示中（確認待ち） | 400〜2000KB/s程度で、分散が小さい安定した帯 |
| 実際に作業中（検索・生成中） | 2500KB/sを大きく超え、かつ激しく変動する |

本実装への移行後、実機で継続的に稼働させて検証したところ、短時間の検証では見えなかった以下の課題が明らかになった。

1. **アイドル時の実測値が閾値に近すぎて頻繁にちらつく問題** — 真のアイドル時でも100〜150KB/s程度まで値が揺れることが判明し、当初の単一閾値(150KB/s)ではその近辺で緑/赤が数秒おきに切り替わってしまった。対策として、閾値をノイズの実測上限から十分離すとともに、状態ごとに異なる閾値を使う**ヒステリシス**を導入した。
2. **メッセージ入力中（未送信）の一時的なI/Oバーストが確認待ちと誤判定される問題** — 入力中にも350KB/s前後までI/O量が上昇するケースが確認された。実際の確認待ちは700〜800KB/sで安定していたため、両者の間に閾値を引き直した。
3. **作業終了直後に一瞬だけ確認待ち（赤）と誤判定される問題** — 生成完了後のI/O減衰（余韻）が確認待ち帯の値を一時的に通過することがあり、送信直後に一瞬赤く点滅する誤判定が発生した。対策として「作業中から確認待ちへは直接遷移させず、必ず一度アイドルを経由する」というルールを追加した（副作用として、作業直後に立て続けに確認が来た場合は一瞬アイドルを経由する仕様となっているが、実用上は許容範囲と判断している）。

これらの調整はいずれも実機での継続的な利用テストを通じて発見・対応したものであり、短時間の検証だけでは見えない実運用特有の課題だった。閾値・プロセス名・ポーリング間隔等は現在 `config.json` として外部化されており、再ビルドなしで調整できる（詳細は [README](../README.md) 参照）。

## ペットモードの追加（2026-07）

Codex Pets（OpenAIがCodexに導入した、エージェントの状態をデスクトップ上のキャラクターアニメーションで伝える機能）に着想を得て、オーバーレイウィンドウの表示形式に「ペット」を追加した。状態判定ロジック（NetworkStateMonitor）には一切手を入れず、状態(-1/0/1/2)を受け取って描画するだけの表示層の追加に限定している。

- 表示形式はトレイメニューから信号機/ペットを切り替え可能。既定は従来どおり信号機（客先・画面共有等でも堅い表示を維持するため）
- ペットは pet.json＋spritesheet.png のフォーマットで定義し、`%LOCALAPPDATA%\ClaudeSignalTray\pets\` に置くことでカスタムペットを追加できる。組み込みペット（ねこ）は初回起動時に同フォルダへ展開され、フォーマットの実例を兼ねる
- 設計の詳細・非採用とした案（タスク内容の吹き出し表示等）は [pet-overlay-design.md](pet-overlay-design.md) を参照

## 得られた知見

- Claudeの自律的な指示遵守（グローバル指示）に、確実性が求められる用途で依存するのはリスクが高い。指示文の表現を工夫しても、実行される保証はないことを複数回の実機検証で確認した。
- Electron/Chromiumベースのデスクトップアプリは、既定では外部のアクセシビリティAPI（UI Automation等）に画面内容を公開しない。外部からの画面監視によるアプローチは、対象アプリがこの種のフレームワークで作られている場合は原則として機能しない。
- 通信内容そのもの（暗号化ペイロード）を見なくても、プロセスのI/O量という間接的な指標だけで、ある程度実用的な状態推定ができる。ただし、短時間の検証だけでは閾値設計の甘さを見落としやすく、実運用に耐える精度に到達するには継続的な実機検証と閾値・ロジックの反復調整が不可欠だった。
