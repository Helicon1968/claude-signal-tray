# claude-signal-tray

Claude Cowork の作業状況（作業中／確認待ちの可能性／待機中）を、Windowsタスクトレイの信号アイコンと常時最前面のオーバーレイウィンドウでひと目で分かるようにする、非公式の常駐アプリです。

> **これは非公式の個人プロジェクトです。Anthropic / Claude とは一切関係ありません。**
> Claude Cowork には外部から状態を取得できる公式APIが無いため、Claudeプロセスの
> I/O量の変化パターンから状態を推定するという、あくまで実用上のヒューリスティックで
> 成り立っています。確実な判定を保証するものではありません。

## これは何のためのツールか

Cowork はバックグラウンドでタスクを実行しますが、確認待ち（承認・入力待ち）になったことに気づきにくい場合があります。本ツールはタスクトレイのアイコンとオーバーレイウィンドウの色を変えることで、ウィンドウを開かなくても一目で状態を把握できるようにします。

- 🟢 緑 = 待機中／完了（何もしなくてよい）
- 🟡 黄 = 作業中
- 🔴 赤（点滅） = 確認待ちの可能性！対応が必要かもしれない
- ⚪ 灰 = Claudeプロセスが見つからない（起動していない等）

## 仕組み（概要）

Windowsの `PerformanceCounter`（`Process` カテゴリの `IO Data Bytes/sec`。ディスク・ネットワーク等のI/O合算値）を使い、`claude` という名前の全プロセスのI/O量を0.7秒間隔でサンプリングし続け、直近数サンプルの中央値から状態を推定します。通信の中身（TLSで暗号化されたペイロード）は一切見ておらず、I/O量という間接的な指標のみを使っています。

3回の手動検証（[experiments/network-probe](experiments/network-probe)）で、以下の傾向が一貫して再現されました。

- 完全アイドル（何も起きていない）: ほぼ0
- AskUserQuestion表示中（確認待ち）: 400〜2000KB/s程度で分散の小さい安定した帯
- 実際に作業中（検索・生成中）: 2500KB/sを大きく超え、かつ激しく変動する

この3段階をもとに緑/黄/赤を切り替えます。実運用での継続テストを通じて、アイドル時のノイズによるちらつきや、入力中・作業終了直後の誤判定など、短時間の検証だけでは見えない課題が見つかり、ヒステリシスの導入や閾値の調整で対応してきました。検討した他のアプローチ（Cowork「グローバル指示」経由でのファイル書き込み、Windows UI Automationによる画面監視、ネットワークパケットキャプチャ）が不採用になった経緯を含む詳しい開発の経緯は [docs/HISTORY.md](docs/HISTORY.md) を参照してください。

## 実行方法

.NET 8 SDK が必要です（ビルド時のみ。ビルド後のexeは自己完結型なので実行するPCに.NETのインストールは不要です）。

```powershell
dotnet publish -c Release
```

生成物:

```
bin\Release\net8.0-windows\win-x64\publish\ClaudeSignalTray.exe
```

実行ファイル自体のアイコン（`icon.ico`、赤/黄/緑の3色バンド）は、実行中にタスクトレイへ
動的に描画される状態ドットとは別物です。エクスプローラーやタスクバーでの表示用です。

> 未署名のexeなので、初回実行時にSmartScreenの警告が出ることがあります。気になる場合はソースからビルドして使ってください。

実行すると、追加設定なしでタスクトレイに丸いアイコンが表示され、Claudeプロセスの監視を開始します。Coworkのグローバル指示への設定は不要です。

### 自動起動（任意）

Windowsの「スタートアップ」フォルダ（`%APPDATA%\Microsoft\Windows\Start Menu\Programs\Startup\`）にショートカットを置くことで、ログオン時に自動起動できます。Windowsサービス化はできません（サービスはセッション0で動作するため、タスクトレイへのUI表示と両立しません）。

**スクリプトで設定する場合（管理者権限不要）:**

```powershell
# 先に dotnet publish -c Release を実行してexeをビルドしておく
powershell -ExecutionPolicy Bypass -File .\scripts\Install-AutoStart.ps1
```

既定のビルド出力パス（`bin\Release\net8.0-windows\win-x64\publish\ClaudeSignalTray.exe`）を自動的に探して、スタートアップフォルダにショートカットを作成します。別の場所に置いたexeを使う場合は `-ExePath` で指定してください。削除する場合は `.\scripts\Uninstall-AutoStart.ps1` を実行します。

> タスクスケジューラ経由の登録も可能ですが、環境（会社ポリシーやセキュリティソフト）によっては
> アクセス拒否になるケースがあったため、より制限を受けにくいスタートアップフォルダ方式を採用しています。

**手動で設定する場合:**

1. `Win + R` → `shell:startup` と入力してEnter（スタートアップフォルダが開きます）
2. `ClaudeSignalTray.exe`（または そのショートカット）をこのフォルダにコピーする

### 旧PowerShell版について

`ClaudeSignalTray.ps1` は初期バージョン（status.jsonベース、Cowork「グローバル指示」への依存が必要）のままのため、現在は実質的に非推奨です。すぐに試したい場合はC#版をビルドしてください。詳細は [docs/HISTORY.md](docs/HISTORY.md) を参照。

## トレイメニューの機能

タスクトレイアイコンを右クリックすると、以下の操作ができます。

| メニュー項目 | 内容 |
| --- | --- |
| 状態表示 | 現在の状態・I/O量・監視プロセス数（クリック不可、表示のみ） |
| デバッグログ出力を開始/停止 | 下記「デバッグ用ログ出力」参照 |
| 設定/ログフォルダを開く | `%LOCALAPPDATA%\ClaudeSignalTray\` をエクスプローラーで開く |
| 設定を再読み込み | `config.json` を編集後、再起動せずに反映する |
| オーバーレイウィンドウを表示 | 下記「オーバーレイウィンドウ」参照 |
| オーバーレイのサイズ（小/大） | オーバーレイウィンドウの表示サイズを切り替える |
| 終了 | アプリを終了する |

## オーバーレイウィンドウ

タスクトレイのアイコンだけでなく、画面上に常に最前面表示される小さなウィンドウ（●＋状態テキスト、例:「確認待ちの可能性 (750 KB/s)」）でも状態を確認できます。

- トレイアイコンを右クリック →「オーバーレイウィンドウを表示」でON/OFFを切り替えます。ON/OFFの状態は`config.json`に保存され、次回起動時も復元されます。
- 初期位置は画面右端・縦中央付近です。ウィンドウ自体をドラッグすると自由に移動でき、移動した位置も`config.json`に保存され、次回起動時に復元されます。
- 「オーバーレイのサイズ」サブメニューから「小」「大」の2種類のサイズを選べます。選択内容も`config.json`（`OverlaySize`: `"Small"`/`"Large"`）に保存され、次回起動時も復元されます。
- タイトルバー・枠なしの小さなウィンドウで、フォーカスを奪わない設定にしているため、他のアプリを操作中でも作業の邪魔になりません（本ツールの目的が「フォアグラウンド作業を妨げないこと」であるため、この点は意図的に配慮しています）。
- 右クリックメニューはトレイアイコンと共通です。

## 設定ファイル（config.json）

閾値やプロセス名などの調整項目は、ソースの書き換え・再ビルドをしなくても `%LOCALAPPDATA%\ClaudeSignalTray\config.json` を編集するだけで変更できます。初回起動時、ファイルが存在しなければ既定値で自動生成されます。

```json
{
  "ProcessName": "claude",
  "PollIntervalMs": 700,
  "RefreshInstancesEveryNTicks": 8,
  "WindowSize": 8,
  "DebounceTicks": 4,
  "IdleEnterBytesPerSec": 250000,
  "ConfirmWaitEnterBytesPerSec": 600000,
  "WorkingThresholdBytesPerSec": 2500000,
  "OverlayEnabled": false,
  "OverlayX": -1,
  "OverlayY": -1,
  "OverlaySize": "Small"
}
```

| 項目 | 内容 |
| --- | --- |
| `ProcessName` | 監視対象プロセス名（既定: `claude`。`claude#1`等の複数インスタンスも自動検出） |
| `PollIntervalMs` | サンプリング間隔(ミリ秒) |
| `RefreshInstancesEveryNTicks` | プロセス一覧の再取得間隔(tick数) |
| `WindowSize` | 中央値算出に使うサンプル数 |
| `DebounceTicks` | 表示切り替えに必要な連続一致回数 |
| `IdleEnterBytesPerSec` | 確認待ち→アイドルに戻る条件（これ以下） |
| `ConfirmWaitEnterBytesPerSec` | アイドル→確認待ちになる条件（これ以上） |
| `WorkingThresholdBytesPerSec` | これ以上なら「作業中」 |
| `OverlayEnabled` / `OverlayX` / `OverlayY` / `OverlaySize` | オーバーレイウィンドウの表示状態・位置・サイズ |

編集後は、タスクトレイアイコンを右クリック →「設定を再読み込み」で、アプリを再起動せずに反映できます。値が壊れている・大小関係が矛盾している場合は、該当項目のみ既定値にフォールバックし、アプリが動かなくなることは避けています。

**別環境での利用について:** 本ツールは通信先のURL/エンドポイントを一切見ておらず、`claude`という名前のプロセス全体のI/O量のみを見ています。そのため、例えば直接Anthropic APIを使う場合とAWS Bedrock経由の場合でプロセス名自体が変わることは基本的にありませんが、ネットワーク経路や応答特性の違いでI/O量の水準が変わる可能性はあります。その場合は上記の閾値を環境に合わせて調整してください。マシンやネットワークが変わる場合は、まず一度デバッグログを取ってから閾値を見直すことをおすすめします。

## デバッグ用ログ出力

タスクトレイアイコンを右クリック →「デバッグログ出力を開始」で、1tick(0.7秒)ごとに生値・中央値・判定結果・確定状態をCSVに記録できます。出力先は `%LOCALAPPDATA%\ClaudeSignalTray\logs\log-{開始時刻}.csv` で、右クリックメニューの「設定/ログフォルダを開く」から直接開けます。

列の意味:

| 列名 | 内容 |
| --- | --- |
| timestamp | 記録時刻 |
| raw_bytes_per_sec | その1tickの生I/O量(平滑化前、バイト/秒) |
| median_bytes_per_sec | 直近ウィンドウの中央値(実際の判定に使う値、バイト/秒) |
| bucket | その時点の分類結果(デバウンス前、-1/0/1/2) |
| status | 画面に表示されている確定状態(デバウンス後、-1/0/1/2) |
| process_count | 監視対象のclaudeプロセス数 |

常時ONにはしていません(ディスク書き込みが不要な場合に増え続けるため)。閾値の再調整や不可解な挙動の分析をしたいときだけ有効にしてください。

## 既知の制限

- **作業中(黄)から確認待ち(赤)へは、必ずアイドル(緑)を経由してから遷移します。** 生成が終わった直後のI/O減衰(余韻)が確認待ち帯の値を一瞬通過することがあり、それをそのまま確認待ちと判定すると送信直後に一瞬赤くなる誤判定が起きるため、意図的にこの制約を入れています。そのため「作業が終わった直後、間を置かず即座に確認を求められる」ケースでは、赤になるまでに数秒余分にかかることがあります。
- **確実性の保証はありません。** 閾値ベースのヒューリスティックなので、環境やClaudeのバージョンによって挙動が変わる可能性があります。
- ディスクI/OとネットワークI/Oを区別していません（`IO Data Bytes/sec` は合算値）。ローカルファイルへ大きな書き込みが発生する場面では誤判定の可能性があります。
- 同じ`ProcessName`に一致する全プロセスのI/O量を合算するため、複数のCowork/Claudeプロセス（例: Claude DesktopとClaude Code CLIを同時起動した場合）を区別できません。片方が確認待ちでも、もう片方の作業でI/O量が上書きされ、作業中(黄)と表示されることがあります（該当プロセスの作業が終われば、合算I/Oが下がり元の表示に戻ります）。
- Windowsのみ対応です（NotifyIcon / WinForms / PerformanceCounterに依存）。
